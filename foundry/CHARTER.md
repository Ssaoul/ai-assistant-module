# Agent Foundry Charter — 에이전트 출산 헌장

> 이 시스템은 **닫힌 에이전트 명단**을 갖지 않는다.
> 필요가 생기면 **에이전트를 만드는 에이전트(Foundry)** 가 새 에이전트를 설계하고,
> **여러 관점이 서로 논쟁(Council)** 한 뒤, **합격점(Gate)** 을 넘은 것만 출산한다.
> 6종 seed는 시작점일 뿐 — 명단은 계속 자란다.

## 1. 왜 이렇게 하는가
고정된 역할 세트는 예측 못 한 필요에 못 따라간다. 대신 **에이전트를 저렴하게, 그러나 함부로는 아니게** 만들 수 있어야 한다. "저렴하게" = 사람이 매번 프롬프트를 손으로 쓰지 않음. "함부로는 아니게" = 합격점을 통과해야만 태어남. 이 둘의 균형이 헌장의 목적이다.

## 2. 출산 파이프라인
```
필요 감지(Need)              누가: 오케스트레이터 / 사람 / 다른 에이전트
   ↓  need-brief 작성
초안(Draft)                  누가: agent-architect (메타 에이전트)
   ↓  agent-spec v0
심의(Council)  ── 병렬 논쟁 ── advocate · skeptic · systems · domain · ops
   ↓  각 관점이 루브릭 채점 + 반대의견
게이트(Gate)                 총점 ≥ 합격점 AND 차단 반대 0  →  통과
   ↓ 통과                          ↓ 미달
출산(Birth)                  재설계 루프 (최대 N회) → 여전히 미달이면 기각(사유 기록)
   ↓
등록(Register)               .claude/agents/<name>.md 작성 + agent-registry.yaml 추가 + ADR
```

## 3. 합격점(Gate) — 정량 기준
각 관점이 아래 5개 축을 0~5로 채점한다. **가중 총점 100점 만점.**

| 축 | 뜻 | 가중 | 차단 조건(하나라도면 자동 기각) |
|---|---|---|---|
| **Necessity** | 기존 에이전트로 안 되는 실제 공백인가 | ×5 | 기존 에이전트와 80%+ 중복 |
| **Clarity** | 목적·입출력·성공기준이 명확한가 | ×4 | 성공기준을 측정 불가 |
| **Safety** | 가드레일·권한 최소성·도메인 경계(PII/SaMD) | ×5 | 가드레일 위반 소지 |
| **Fit** | 키트·자율등급·기존 워크플로에 맞는가 | ×3 | 저장소 autonomy 등급 초과 요구 |
| **Cost** | 토큰/유지비 대비 가치 | ×3 | 가치 대비 상시비용 과다 |

- **합격점 = 가중 총점 75 / 100 이상**, 그리고 **차단 조건 0건**, 그리고 **skeptic이 낸 반대 중 미해결 blocking 0건**.
- 합격점은 저장소 위험도에 따라 조정: `risk: high`(L0) 저장소는 **85점**으로 상향.

## 4. Council — 관점(Perspective)들
심의체는 최소 아래 5관점을 포함한다. 관점 자체도 `perspectives.yaml`에서 추가/교체 가능(= 심의체도 자란다).

- **advocate** — 이 에이전트가 푸는 문제의 가치를 최대한 강하게 변론. 초안이 그 가치를 실제로 달성하는지 확인.
- **skeptic (red-team)** — 중복·범위확장(scope creep)·실패모드·악용 가능성을 공격. *기본값은 "반대"*, 설득돼야 물러선다.
- **systems** — 키트 적합성, 도구 권한 최소화, 다른 에이전트와의 인터페이스, 유지보수성.
- **domain** — 휴레이 도메인·규제 경계(개인정보, SaMD 진단/처방 금지, 컨소시엄 규약) 준수.
- **ops** — 상시 운영비(토큰), 관측가능성, 롤백 용이성, 자율등급 정합성.

논쟁은 **최소 1라운드, 최대 3라운드**. 라운드마다 architect가 반대의견을 반영해 초안을 갱신하고 다시 채점. 3라운드 후에도 미달이면 기각.

## 5. 출산 후 규약
- 새 에이전트는 `status: probation`(수습)으로 태어난다. 수습 중엔 자율등급이 한 단계 낮게 강제된다(예: L2 저장소여도 L1처럼 PR 필수).
- **승격(probation → active)**: 수습 기간 동안 실제 작업 N건에서 문제 없고, 사람이 승인하면 active. ADR 기록.
- **폐기(deprecate)**: 90일간 미사용 or 상위 에이전트에 흡수되면 registry에서 `deprecated` 표시 후 아카이브.

## 6. 절대 규칙
1. 에이전트는 **Foundry를 거치지 않고 태어날 수 없다.** `.claude/agents/`에 직접 수기 추가 금지(seed 제외, seed도 소급 등록).
2. **에이전트가 에이전트를 만들 수 있으나, 게이트를 스스로 낮출 수 없다.** 합격점·가중치·차단조건 변경은 사람 승인 + ADR.
3. 새 에이전트의 **도구 권한은 최소로.** 필요 입증 안 된 도구는 부여 금지(systems 관점이 강제).
4. 심의 과정(초안·채점·반대·결정)은 **`foundry/ledger/`에 전문 기록** — 왜 태어났/기각됐는지 추적 가능해야 한다.
